# Proof of Delivery

**Proof of Delivery (POD)** is evidence captured by the driver at the point of delivery, confirming that an order — or a specific item within an order — was successfully handed over. Fleet-Ops supports four POD methods, and the required method is configured per Order Configuration.

## POD Methods

| Method | How it works |
| :--- | :--- |
| **Signature** | Recipient signs on the driver's phone screen using the Navigator app |
| **Photo** | Driver takes a photo of the delivered items or the recipient |
| **QR Code** | Driver scans a QR code printed on the parcel or handed to the recipient |
| **Custom Data** | Driver enters free-form data (e.g., a pin code, reference number) |

<Callout type="info">
POD requirements are set per Order Configuration. You can require POD for the entire order or independently for each waypoint and entity. See [Order Configurations](/docs/fleet-ops/operations/order-configurations/overview) for setup.
</Callout>

## Configuring POD in Order Configurations

POD is configured inside each Order Configuration, not on individual orders. To set it up:

<Steps>
  <Step>
    Navigate to **Fleet-Ops → Operations → Order Configurations**.
  </Step>
  <Step>
    Open or create an Order Configuration.
  </Step>
  <Step>
    In the configuration editor, set:
    - **POD Required** — toggle on to require proof before order completion
    - **POD Method** — select Signature, Photo, QR Code, or Custom Data
  </Step>
  <Step>
    For multi-stop orders, POD can also be set independently per activity step in the **Activity Flow** editor.
  </Step>
  <Step>
    Save the configuration. All new orders using this config will require the selected POD method.
  </Step>
</Steps>

## How Drivers Capture POD

In the Navigator app, when a driver arrives at the final delivery stop:

1. The app prompts them to capture POD based on the configured method
2. For **signature**: the recipient signs directly on screen
3. For **photo**: the app opens the camera; the driver photographs the delivery
4. For **QR code**: the app opens the scanner; the driver scans the parcel QR
5. For **custom data**: the driver enters the required text or numeric value
6. The driver taps **Complete** — the POD is uploaded and the order closes

## POD for Entities and Waypoints

For orders with multiple items (entities) or multiple stops (waypoints), POD can be captured individually:

- **Per entity** — each item requires its own proof (e.g., photo of each parcel delivered)
- **Per waypoint** — each stop requires POD before the driver can move to the next stop

This creates a granular audit trail for complex multi-drop deliveries.

## Viewing POD in the Console

Captured POD appears in the **Order Detail Panel** under the **Proofs** section:

- Signature images are displayed inline
- Photos are displayed as thumbnails with full-screen view
- QR scan records show the scan result and timestamp
- Custom data entries show the captured value

Each proof record includes:
- Timestamp of capture
- Driver location at time of capture
- The subject (order, waypoint, or entity)
- Optional remarks entered by the driver

## Downloading POD

From the order detail panel, click **Download** on any proof to save the signature image or photo. For bulk exports, use the order export function — CSV exports include proof URLs that link to the stored files.

## POD via the API

POD can also be captured via the API. Three endpoints accept proof submissions:

```bash
POST /v1/orders/{id}/capture-signature/{subjectId}
POST /v1/orders/{id}/capture-photo/{subjectId}
POST /v1/orders/{id}/capture-qr/{subjectId}
```

The `subjectId` is the UUID of the order, entity, or waypoint to attach the proof to.

See the [API Reference](/docs/api) for request body schemas.
