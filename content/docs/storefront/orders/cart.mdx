# Cart

The **Cart** is the first step in the checkout process. It acts as a temporary holding area for the items a customer intends to purchase.

In Storefront, Carts are persisted in the database. This means that if a customer adds items to their cart on their mobile phone, they can log into your web application later and see the exact same items waiting for them.

## Creating a Cart

When a customer first interacts with your frontend application, you should use the [Storefront API](/docs/api-reference/storefront) to create a new Cart or retrieve their existing one.

If the customer is authenticated (logged in), the API will automatically return their saved Cart. If they are a guest, the API will create a new, anonymous Cart and return a unique `cart_id`. Your frontend application must store this `cart_id` (e.g., in local storage) to reference the Cart in subsequent requests.

## Adding Items to the Cart

When a customer clicks "Add to Cart", your frontend application sends a request to the API with the following information:

- The `product_id` of the item.
- The `quantity` desired.
- The `variant_id` (if the product has [Variants](/docs/storefront/catalog/variants-and-addons)).
- An array of `addon_ids` (if the customer selected any optional [Add-ons](/docs/storefront/catalog/variants-and-addons)).

Storefront validates the request (ensuring the product is active and the store is open) and adds the item to the Cart as a new "Line Item".

## Cart Calculations

One of the primary benefits of using Storefront is that it handles all the complex pricing math for you.

Every time you add, update, or remove an item from the Cart, the API recalculates the totals and returns the updated Cart object.

The Cart object includes:
- **Subtotal**: The sum of all line items (Base Price + Variant Price + Add-on Prices) multiplied by their quantities.
- **Discount**: The total amount deducted by any applied [Promotions](/docs/storefront/promotions).
- **Taxes**: Calculated based on the Store's tax settings.
- **Total**: The final amount the customer needs to pay (Subtotal - Discount + Taxes).

*Note: The delivery fee is not calculated during the Cart phase. It is calculated during the [Checkout](/docs/storefront/orders/checkout) phase once the customer provides their delivery address.*

## Multi-Store Carts

If you are operating a multi-vendor [Network](/docs/storefront/stores/stores-and-networks) (like a food delivery app), a common question is whether a customer can add items from multiple different Stores into a single Cart.

By default, Storefront **does not** support multi-store carts.

If a customer has items from "Store A" in their cart and attempts to add an item from "Store B", the API will return an error. Your frontend application should catch this error and prompt the customer: "You already have items from another store in your cart. Would you like to clear your cart and start a new order with this store?"

This restriction exists because each Store operates independently, with its own location, operating hours, and dispatch requirements. Fulfilling a single order from multiple physical locations simultaneously is logistically complex and generally avoided in on-demand delivery models.
