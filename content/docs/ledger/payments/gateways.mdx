# Payment Gateways

**Payment Gateways** are the external services that process real money. Ledger integrates with Stripe, QPay, and Cash — each gateway stores encrypted credentials and supports sandbox mode for testing.

Navigate to **Ledger → Payments → Gateways**.

![Gateway list — gateway name, driver, environment badge, status, and configuration button](/images/placeholder.png)

## Supported Gateways

| Gateway | Driver | Use Case |
| :--- | :--- | :--- |
| **Stripe** | `stripe` | Card payments, refunds, setup intents globally |
| **QPay** | `qpay` | Invoice-based payments for Mongolia and regional markets |
| **Cash** | `cash` | Manual/offline payments; no real processing |

## Adding a Gateway

<Steps>
  <Step>
    Navigate to **Ledger → Payments → Gateways**.
  </Step>
  <Step>
    Click **Add Gateway**.
  </Step>
  <Step>
    Enter a **Name** and select the **Driver** (stripe, qpay, or cash).
  </Step>
  <Step>
    Enter the gateway credentials (see per-driver configuration below).
  </Step>
  <Step>
    Set **Environment** to `sandbox` while testing, `live` for production.
  </Step>
  <Step>
    Click **Save**.
  </Step>
</Steps>

## Stripe Configuration

| Field | Description |
| :--- | :--- |
| **Publishable Key** | `pk_test_...` or `pk_live_...` |
| **Secret Key** | `sk_test_...` or `sk_live_...` |
| **Webhook Secret** | `whsec_...` — from Stripe Dashboard → Webhooks |
| **Environment** | sandbox or live |

Register Ledger's webhook URL in the Stripe Dashboard so Stripe can notify Fleetbase of `charge.succeeded`, `charge.failed`, and `charge.refunded` events.

The webhook URL is shown in the gateway detail: `POST /ledger/webhooks/stripe`

## QPay Configuration

| Field | Description |
| :--- | :--- |
| **Username** | QPay merchant username |
| **Password** | QPay merchant password |
| **Invoice Code** | QPay invoice type code |
| **Environment** | sandbox or live |

## Cash Configuration

No credentials required. Name the gateway (e.g., "Cash Payment") and save. Cash gateway processes are immediate and always succeed — use for manual payment recording or testing.

## Gateway Capabilities

Each gateway exposes the operations it supports:

| Capability | Description |
| :--- | :--- |
| `purchase` | Process a charge |
| `refund` | Issue a refund |
| `setup_intent` | Tokenize a payment method without charging |

Check capabilities before calling an operation. The Cash driver supports `purchase` only.

## Security

Gateway credentials are:
- Stored **encrypted at rest** (AES-256)
- **Never exposed** in API responses
- Decrypted only in memory when a payment operation is executed

## Gateway Transaction Audit Log

Every interaction with a gateway (charge, refund, webhook) is logged as a `GatewayTransaction`. View the log for any gateway:

1. Open the gateway in Ledger
2. Click the **Transactions** tab

The log includes status, amount, gateway reference ID, raw response, and processing timestamp — useful for debugging payment issues.
