# Request Logs

The **Request Logs** panel in the Developer Console provides a complete, searchable history of all API requests made to your Fleetbase instance. This is an essential tool for monitoring API usage, debugging integrations, and auditing system activity.

## Viewing Request Logs

To access the request logs:

1. Navigate to **Developers** > **Request Logs** in the Fleetbase Console.
2. The panel displays a chronological list of all API requests, including:
   - **Method**: The HTTP method used (GET, POST, PUT, DELETE).
   - **Endpoint**: The specific API route that was called.
   - **Status**: The HTTP response status code (e.g., 200 OK, 400 Bad Request, 500 Internal Server Error).
   - **Date & Time**: When the request was received.
   - **API Key**: The specific API key used to authenticate the request (if applicable).

## Inspecting a Request

Clicking on a specific request in the list opens a detailed view, providing all the information needed to debug an issue.

### Request Details

The detailed view includes:

- **Headers**: The HTTP headers sent with the request, including `Authorization`, `Content-Type`, and `User-Agent`.
- **Payload**: The JSON body sent with the request (for POST, PUT, and PATCH requests).
- **Query Parameters**: Any URL parameters included in the request.

### Response Details

The detailed view also includes the response generated by Fleetbase:

- **Status Code**: The final HTTP status code returned to the client.
- **Response Body**: The JSON payload returned by the API. This is particularly useful for inspecting error messages or validation failures.
- **Duration**: The time taken (in milliseconds) for the server to process the request and generate a response.

## Filtering and Searching

When debugging a specific issue, you can use the search and filter tools to quickly find relevant requests.

- **Search by Endpoint**: Enter a specific route (e.g., `/fleet-ops/v1/orders`) to see all requests made to that endpoint.
- **Filter by Status**: Filter the list to show only failed requests (e.g., 4xx or 5xx status codes).
- **Filter by API Key**: If you suspect an issue with a specific integration, you can filter the logs to show only requests authenticated with that integration's API key.

## Test vs. Live Logs

Remember that Request Logs are scoped to the current environment. If you are in **Test Mode**, you will only see requests made using Test API keys. To view production traffic, switch to **Live Mode** using the toggle in the top navigation bar.

## Data Retention

To ensure optimal database performance, Request Logs are typically retained for a limited period (e.g., 30 days) before being automatically purged. If you require long-term auditing, consider exporting the logs or configuring a webhook to send critical events to an external logging service.
